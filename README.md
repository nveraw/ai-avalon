# Avalon — AI-Powered Resistance

A single-player implementation of [The Resistance: Avalon](<https://en.wikipedia.org/wiki/The_Resistance_(game)>) where all bot players are driven by real AI agents (Mistral Large via LangChain). The human plays against fully autonomous agents that reason, deceive, coordinate, and remember across the entire game.

---

## Tech Stack

| Layer             | Technology                             |
| ----------------- | -------------------------------------- |
| Frontend          | React + TypeScript + Vite              |
| Styling           | Tailwind CSS v4                        |
| Backend / API     | Next.js App Router (route handlers)    |
| AI model          | Mistral Large (`mistral-large-latest`) |
| AI framework      | LangChain (`@langchain/mistralai`)     |
| Structured output | Zod                                    |
| Session store     | In-memory (Redis-ready)                |

# Getting Started

### Prerequisites

- Node.js 18+
- A [Mistral AI](https://console.mistral.ai/) API key

### Install

```bash
npm install
```

### Environment

Create a `.env.local` file in the project root:

```env
MISTRAL_API_KEY=your_key_here
```

### Run

```bash
npm run dev
```

Open [http://localhost:3000](http://localhost:3000).

## How It Works

### Game flow

```
Lobby → Night → Board → Team Selection → Voting → Quest → (repeat) → Assassination → Results
```

The human is always `players[0]`. All other players are AI agents. Each game phase maps to one API call.

## Role assignment

Roles are assigned server-side on `/init`. The client never receives bot roles — only the human player's role and night knowledge are returned. Bot roles are stored in `GameState` on the server and never serialised into API responses, so they cannot be inspected in the browser network tab.

### AI agent context

Every AI agent receives a layered system prompt built in `lib/agents/prompts`:

1. **Role strategy** — role-specific goals, win conditions, and multi-stage tactics (early/mid/late game)
2. **Suspicion model instructions** — how to maintain and update per-player evil probability estimates, seeded from night knowledge
3. **Decision framework** — a 6-step process (read state → check win conditions → update suspicion → evaluate team → decide → craft message)
4. **Public game state** — current round, score, leader, rejection streak, quest history
5. **Game summary** — compressed recap of previous rounds generated by a neutral summariser agent after each quest
6. **Recent chat** — last 12 messages for immediate conversational context
7. **Private memory** — the agent's own log of previous decisions and reasoning (last 20 entries)

### Structured outputs

All AI decisions use `model.withStructuredOutput(schema)` via Zod schemas defined in `lib/output.schema.ts`. Each schema separates `privateReasoning` (strategic analysis, never returned to the client) from `publicMessage` (what the player says at the table).

## Known Limitations

- **In-memory store** does not survive server restarts. Use Redis for persistence.
- **No authentication** — Do not expose to untrusted networks without adding auth.
- **AI response latency** — voting and quest phases call all bots in parallel (`Promise.all`) but each call still takes 1–3 seconds depending on Mistral API load.
- **5-player games only have 2 Evil players** — the AI is tuned for this ratio but may play sub-optimally in 9–10 player games with more Evil agents until further prompt tuning.
